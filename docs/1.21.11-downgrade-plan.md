# Minecraft 1.21.11 Downgrade Plan

Plan for the downgrade branch that targets Minecraft
1.21.11 while keeping the current 26.1+ branch easy to merge downstream.

## Goals

- Target Minecraft 1.21.11 first.
- Keep all current artifacts in scope:
  - Veinminer Paper plugin
  - Veinminer Fabric mod
  - Veinminer NeoForge mod
  - Veinminer Client Fabric addon
  - Veinminer Client NeoForge addon
  - Veinminer Enchantment addon
  - datapacks
- Prefer build-chain and compatibility-adapter changes over broad rewrites of
  the vein mining implementation.
- Keep Outlet in the publishing workflow. Do not replace Outlet with hardcoded
  Modrinth game-version lists.

## Current Baseline

The main branch targets Minecraft 26.1+, Java 25, and unobfuscated Minecraft
artifacts. The shared and loader-specific code already uses 26.1 API names such
as `net.minecraft.resources.Identifier` and newer client rendering classes.

The downgrade branch must account for the 1.21.11 boundary:

- 1.21.11 is still an obfuscated Minecraft release.
- 26.1 is the first unobfuscated release.
- Fabric and Paper can compile against Mojang mappings for 1.21.11.
- NeoForge has a 21.11 line for Minecraft 1.21.11 and uses Mojang names in the
  development environment.

Useful references:

- Paper userdev 1.20.5+ mapping behavior and Mojang production output:
  <https://docs.papermc.io/paper/dev/userdev/>
- Paper 1.21.11 notes on 26.1 unobfuscated runtime changes:
  <https://papermc.io/news/1-21-11/>
- Fabric mapping migration docs:
  <https://docs.fabricmc.net/develop/migrating-mappings/loom>
- Fabric 1.21.11 notes:
  <https://fabricmc.net/2025/12/05/12111.html>
- NeoForge 21.11 release notes:
  <https://neoforged.net/news/21.11release/>
- 1.21.11 Mojang mapping browser:
  <https://mappings.dev/1.21.11/>

## Outlet And Version Properties

Keep Outlet as the mechanism that expands supported Minecraft versions for
publishing. The downgrade branch should adapt the properties that feed
`outlet.mcVersionRange`.

Use this first-pass property shape in `gradle.properties`:

```properties
gameVersion=1.21.11
enchantmentVersions=1.21.11

fabricSupportedVersions=1.21.11
neoforgeSupportedVersions=1.21.11
paperSupportedVersions=1.21.11
```

The existing usage pattern should stay intact:

- `buildSrc/src/main/kotlin/fabric-script.gradle.kts` reads
  `fabricSupportedVersions`.
- `veinminer/fabric/build.gradle.kts` and
  `veinminer-client/fabric/build.gradle.kts` call `outlet.mcVersions()`.
- `veinminer/neoforge/build.gradle.kts` and
  `veinminer-client/neoforge/build.gradle.kts` read
  `neoforgeSupportedVersions`.
- `veinminer/paper/build.gradle.kts` reads `paperSupportedVersions`.
- `veinminer-enchant/build.gradle.kts` reads `enchantmentVersions`.

If Outlet resolves a 26.1 dependency for a 1.21.11 branch, pin the affected
loader dependency directly while still keeping Outlet for published Minecraft
version metadata.

Expected dependency baselines:

```properties
paperVersion=1.21.11-R0.1-SNAPSHOT
neoforgeVersion=21.11.31-beta
fancyModLoaderVersion=<verify compatible 21.11 FML version>
fabricPermAPI=<verify compatible 1.21.11 version>
```

Use the newest verified `21.11.x-beta` NeoForge if `21.11.31-beta` is no longer
the right choice when the branch is implemented.

## Build-Chain Changes

### Java And Kotlin

Minecraft 1.21.11 uses Java 21. Change the branch toolchain from Java 25 to
Java 21:

- `java.toolchain.languageVersion = JavaLanguageVersion.of(21)`
- `compileJava.options.release = 21`
- Kotlin `jvmTarget = JvmTarget.JVM_21`
- Fabric metadata `"java": ">=21"`

Keep the Kotlin compiler and serialization plugin versions unchanged unless
Gradle or a loader plugin forces a downgrade.

### Fabric

For every Loom-backed module, compile against official Mojang mappings:

```kotlin
dependencies {
    minecraft("com.mojang:minecraft:$gameVersion")
    mappings(loom.officialMojangMappings())
}
```

Apply this to:

- `core`
- `veinminer/common`
- `veinminer/fabric`
- `veinminer-client/common`
- `veinminer-client/fabric`

Keep access wideners in the `official` namespace. If descriptors fail to
compile, run Loom's class tweaker migration task or manually update descriptors
against `mappings.dev/1.21.11`.

### Paper

Keep Paper as Mojang production output:

```kotlin
paperweight.reobfArtifactConfiguration =
    io.papermc.paperweight.userdev.ReobfArtifactConfiguration.MOJANG_PRODUCTION
```

Do not make `reobfJar` the release artifact for this branch. The Paper artifact
should remain Mojang-mapped and Paper-only.

### NeoForge

Keep the existing ModDevGradle structure if it resolves with NeoForge 21.11.
Set `neoforgeVersion` to a verified 21.11 line and update KotlinLangForge only
if the current dependency requires 26.1.

Expected first failure areas are access transformer descriptors and client
rendering API changes.

## Source Compatibility Work

Start with build-only changes, then repair compile errors in narrow areas.
Avoid refactoring the shared vein mining algorithm unless a 1.21.11 API change
forces it.

Expected repair zones:

- Resource IDs:
  - Current code already uses `Identifier`, which is valid in 1.21.11 Mojang
    mappings.
  - Below 1.21.11 will likely need `ResourceLocation` compatibility, but that is
    outside the first milestone.
- Client rendering:
  - `GuiGraphicsExtractor`
  - `RenderPipelines`
  - `RenderSetup`
  - `RenderTypes`
  - `LevelRenderer.renderBlockOutline` mixin signature
- Custom payload networking:
  - `CustomPacketPayload`
  - `StreamCodec`
  - `RegistryFriendlyByteBuf`
  - Fabric and NeoForge registration methods
- Permissions:
  - `net.minecraft.server.permissions.*`
  - loader-specific permission fallback behavior
- Mining speed modifier:
  - `AttributeModifier` constructor and operation names
  - equipment slot damage signatures
- Access widening and transforming:
  - `ServerPlayer.server`
  - `ServerCommonPacketListenerImpl.connection`
  - client render pipeline internals

If compatibility logic is needed, add small branch-local adapters instead of
spreading version checks through gameplay code. Good adapter candidates:

- client HUD/render bridge
- packet payload registration bridge
- permission check bridge
- ID construction helper

## Below 1.21.11

Do not include below-1.21.11 support in the first milestone.

After 1.21.11 builds and passes runtime smoke tests, evaluate a second milestone
for 1.21.1. That milestone should probably introduce version-specific
source sets or facades because Mojang names before 1.21.11 differ more
significantly, especially around `ResourceLocation` versus `Identifier`.

## Verification Checklist

Run build checks:

```bash
./gradlew clean build
./gradlew :veinminer:veinminer-paper:shadowJar
./gradlew :veinminer:veinminer-fabric:build
./gradlew :veinminer:veinminer-neoforge:build
./gradlew :veinminer-client:veinminer-client-fabric:build
./gradlew :veinminer-client:veinminer-client-neoforge:build
./gradlew :veinminer-enchant:build
```

Run runtime smoke tests:

- Paper 1.21.11:
  - plugin loads
  - `/veinminer` command works
  - config reload works
  - vein mining works in survival
  - Folia task still starts if supported
- Fabric 1.21.11:
  - server mod loads
  - client addon loads
  - networking handshake completes
  - hotkey changes shape/depth
  - preview overlay renders
- NeoForge 1.21.11:
  - server mod loads
  - client addon loads
  - networking handshake completes
  - permission fallback works
  - preview overlay renders
- Enchantment addon:
  - Paper loader path works
  - Fabric loader path works
  - NeoForge loader path works
  - enchantment requirement gates vein mining correctly

Verify generated metadata:

- Fabric JSON declares Minecraft `1.21.11` and Java `>=21`.
- NeoForge TOML declares Minecraft `1.21.11` and a 21.11 NeoForge range.
- Paper plugin metadata remains API version `1.21`.
- Modrinth game versions are generated through Outlet and contain only
  `1.21.11` for the first downgrade release.

## Maintainability Rules

- Keep branch-only build changes centralized in `gradle.properties` and
  `buildSrc` whenever possible.
- Keep Outlet in use for publishing metadata.
- Keep gameplay logic changes minimal and evidence-driven.
- Prefer small adapters over broad `if version` branches.
- Document every unavoidable API shim in the file where it is introduced.
- Re-run the full verification checklist after merging updates from main into
  the downgrade branch.
